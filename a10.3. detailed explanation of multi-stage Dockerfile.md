## 🎯 Part 1: Dockerfile কি?

### সহজ ভাষায়:

```
কল্পনা করুন: নতুন কম্পিউটার কিনলেন

Manual Setup:
1. Windows install করুন
2. Node.js install করুন
3. আপনার app এর dependencies install
4. Code copy করুন
5. App run করুন

প্রতিবার এই কাজ করতে হবে! 😓

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Dockerfile = Recipe (রেসিপি)

একবার লিখুন:
- কোন OS লাগবে
- কি কি software লাগবে
- কিভাবে app চালাবেন

Docker এটা পড়ে automatic setup করবে! ✓

যেকোনো কম্পিউটারে একই environment! 🎯
```

### Network Layer এর সাথে তুলনা:

```
Network Layer:
- IP Packet = instructions (কোথায় যাবে)
- Router = processor (instructions execute)
- Destination = final setup

Docker:
- Dockerfile = instructions (কিভাবে build করবে)
- Docker Engine = processor (instructions execute)
- Image = final setup (ready to run)

উভয়ই step-by-step instructions follow করে!
```

## 🏗️ Part 2: Multi-stage Build কি?

### সমস্যা: Single-stage Build

```dockerfile
# ❌ সমস্যাযুক্ত approach

FROM node:22
WORKDIR /app
COPY . .
RUN yarn install    # সব dependencies
RUN yarn build      # Build করা

CMD ["node", "dist/main"]

Final Image Size: 1.5 GB! 🐘
```

**সমস্যা:**

```
Build করার জন্য যা লাগে:
📦 TypeScript compiler
📦 Testing tools
📦 Build tools
📦 Development dependencies
Total: 800 MB

Run করার জন্য যা লাগে:
📦 Compiled JavaScript
📦 Production dependencies only
Total: 200 MB

কিন্তু single-stage এ সব কিছু final image এ থাকে!
Waste: 600 MB! 💸
```

### সমাধান: Multi-stage Build

```
Network Layer এর Fragmentation এর মতো:

Big Packet (1500 bytes):
→ Fragment 1 (500 bytes)
→ Fragment 2 (500 bytes)
→ Fragment 3 (500 bytes)

Destination এ শুধু data নেয়, header ফেলে দেয়

Multi-stage Build:
Big Build Process:
→ Stage 1: Build everything
→ Stage 2: Install prod deps
→ Stage 3: শুধু final files নেয়

Final image এ শুধু run করার জন্য যা লাগে! ✓
```

## 📦 Part 3: প্রতিটি Step বিস্তারিত

### Stage 1: Builder (বিল্ডার)

```dockerfile
FROM node:22-slim AS builder
WORKDIR /home/node/app
COPY . .
RUN yarn install && yarn build
```

#### Line-by-line ব্যাখ্যা:

```dockerfile
FROM node:22-slim AS builder
```

**কি হচ্ছে:**
```
FROM = Base image নির্বাচন করা

node:22-slim = 
- Node.js version 22
- Slim variant (ছোট size)
- Linux OS included

AS builder = 
- এই stage এর নাম "builder"
- পরে reference করার জন্য

Network analogy:
FROM = কোন network protocol ব্যবহার করবেন?
```

**বিকল্প:**
```dockerfile
FROM node:22           # Full version (বড়, সব tools আছে)
FROM node:22-slim      # Smaller (unnecessary tools removed)
FROM node:22-alpine    # Smallest (Alpine Linux, 40 MB!)
```

---

```dockerfile
WORKDIR /home/node/app
```

**কি হচ্ছে:**
```
WORKDIR = কাজের directory set করা

/home/node/app = 
- এই folder এ সব কাজ হবে
- না থাকলে create করবে
- যেমন: cd /home/node/app

Linux এ folder structure:
/
├── home/
│   └── node/
│       └── app/     ← এখানে কাজ
│           ├── package.json
│           ├── src/
│           └── dist/
```

**কেন /home/node/app?**
```
Security:
- Root user (/) use করা dangerous
- node = non-root user
- Limited permissions ✓
```

---

```dockerfile
COPY . .
```

**কি হচ্ছে:**
```
COPY source destination

প্রথম . = আপনার local computer (host)
দ্বিতীয় . = container এর WORKDIR

Example:
Host:
/my-project/
├── package.json
├── src/
│   └── main.ts
├── tsconfig.json
└── Dockerfile

Container এ copy হবে:
/home/node/app/
├── package.json
├── src/
│   └── main.ts
└── tsconfig.json

Network analogy:
COPY = Data transfer
Host → Container (like PC → Server)
```

**কি কি copy হয় না:**
```
.dockerignore file থাকলে:

# .dockerignore
node_modules/
dist/
.git/
*.log

এগুলো copy হবে না ✓
(Bandwidth save, like compression!)
```

---

```dockerfile
RUN yarn install && yarn build
```

**কি হচ্ছে:**
```
RUN = Container এর ভিতরে command execute

yarn install:
- package.json পড়ে
- সব dependencies download করে
- node_modules/ folder এ রাখে

&& = এবং (first success হলে second চলবে)

yarn build:
- TypeScript compile করে
- src/ folder থেকে dist/ এ output
- Production-ready JavaScript তৈরি

Timeline:
T=0: yarn install শুরু
T=30s: Dependencies downloaded
T=31s: yarn build শুরু
T=45s: TypeScript compiled ✓
```

**কেন && ব্যবহার করা:**
```
Option 1 (❌):
RUN yarn install
RUN yarn build
= 2টা layers, বেশি space

Option 2 (✓):
RUN yarn install && yarn build
= 1টা layer, কম space

Docker layers = Network packets এর মতো
কম layers = efficient!
```

**এই Stage এর পরে:**
```
/home/node/app/
├── node_modules/       ← ALL dependencies (dev + prod)
├── src/                ← Original TypeScript
├── dist/               ← Compiled JavaScript ✓
├── package.json
└── tsconfig.json

Size: ~1.2 GB
```

### Stage 2: Production Builder

```dockerfile
FROM node:22-slim AS prod-builder
WORKDIR /home/node/app
ENV NODE_ENV=production
COPY package.json yarn.lock ./
RUN yarn install
```

#### Line-by-line ব্যাখ্যা:

```dockerfile
FROM node:22-slim AS prod-builder
```

**কি হচ্ছে:**
```
নতুন clean stage শুরু!

Stage 1 থেকে কিছু নেই
Fresh start ✓

Network analogy:
নতুন router = previous router এর state নেই
```

---

```dockerfile
ENV NODE_ENV=production
```

**কি হচ্ছে:**
```
ENV = Environment variable set করা

NODE_ENV=production মানে:
- Development dependencies skip করবে
- Production optimization চালু
- Debug mode off

yarn install দেখবে:
package.json:
{
  "dependencies": {
    "express": "^4.18.0"    ← Install করবে ✓
  },
  "devDependencies": {
    "typescript": "^5.0.0",  ← Skip করবে ✗
    "jest": "^29.0.0"        ← Skip করবে ✗
  }
}

Size reduce: ~400 MB! 💰
```

---

```dockerfile
COPY package.json yarn.lock ./
```

**কি হচ্ছে:**
```
শুধু 2টা file copy করা:

package.json = কি কি package লাগবে
yarn.lock = exact versions (consistency)

কেন সব file copy না করে শুধু এই 2টা?

Docker Layer Caching:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Build 1:
COPY package.json    ← Execute
RUN yarn install     ← Execute (5 minutes)

Code পরিবর্তন করলাম (src/main.ts)

Build 2:
COPY package.json    ← Cached! (unchanged)
RUN yarn install     ← Cached! (0 seconds) ✓

কিন্তু যদি COPY . . করতাম:
Code পরিবর্তন → COPY layer invalid
→ yarn install আবার run (5 minutes) 😓

Best Practice: যা কম পরিবর্তন হয় তা আগে copy!
```

```dockerfile
RUN yarn install
```

**কি হচ্ছে:**
```
শুধু production dependencies install

NODE_ENV=production দেখে yarn:

Installing:
✓ express

```

```
└── yarn.lock

Size: ~200 MB
```

### Stage 3: Final Runtime (চূড়ান্ত)

```dockerfile
FROM gcr.io/distroless/nodejs22

ARG PORT
ENV NODE_ENV=production
WORKDIR /home/node/app

COPY --from=prod-builder /home/node/app/node_modules ./node_modules
COPY --from=builder /home/node/app/package.json .
COPY --from=builder /home/node/app/dist ./dist

EXPOSE ${PORT}
CMD ["--enable-source-maps", "dist/src/main"]
```

#### Line-by-line ব্যাখ্যা:

```dockerfile
FROM gcr.io/distroless/nodejs22
```

**কি হচ্ছে:**
```
Distroless = খুবই minimal base image

Normal node:22-slim:
✓ Node.js runtime
✓ npm/yarn
✓ Shell (bash)
✓ Package managers (apt)
✓ Utilities (curl, wget)
Size: 180 MB

Distroless:
✓ Node.js runtime
✗ NO shell
✗ NO package managers
✗ NO utilities
Size: 120 MB

Security benefits:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

With shell:
Hacker → Container access → Run commands! 😱
cat /etc/passwd
rm -rf /
wget malware.com/virus.sh

Without shell:
Hacker → Container access → Can't run anything! ✓
No bash, no commands!

Attack surface reduced by 80%! 🛡️
```

**Network Security analogy:**
```
Full OS = Router with telnet, SSH, HTTP, FTP
Distroless = Router শুধু routing করে, nothing else

Less services = Less vulnerabilities
```

```dockerfile
ARG PORT
```

**কি হচ্ছে:**
```
ARG = Build-time variable

Docker build করার সময় দিতে পারেন:
docker build --build-arg PORT=3000 -t myapp .

বা না দিলে default নেবে (Dockerfile এ থাকলে)

ARG vs ENV:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

ARG PORT
- Build সময়ে ব্যবহার
- Final container এ নেই
- Use: EXPOSE ${PORT}

ENV PORT=3000
- Runtime এ আছে
- App code এ পাবে: process.env.PORT
- পরিবর্তন করা যায়: docker run -e PORT=4000
```

**কি হচ্ছে:**

```
--from=prod-builder = Stage 2 থেকে copy করা

শুধু production dependencies নিয়ে আসছি! ✓

Multi-stage benefit:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

node_modules/ = 150 MB (prod only)

Copy from Stage 2 = 150 MB ✓

Multi-stage → শুধু needed files নেয়, rest ফেলে
```

**কি হচ্ছে:**
```
Container start হলে কোন command চালাবে

Full command:
node --enable-source-maps dist/src/main

Breakdown:
node                  ← Node.js executor
--enable-source-maps  ← Flag (debug সহজ করে)
dist/src/main         ← Entry point file

Source Maps কি:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

TypeScript (src/main.ts):
async function getUser() {  ← Line 15
  throw new Error("Bug!");
}

Compiled (dist/src/main.js):
function getUser(){return __awaiter(this,void 0,void 0,function*(){
throw new Error("Bug!");})}  ← Line 1 (unreadable!)

Error without source maps:
Error at Line 1 in main.js 😕
কোথায় ভুল? বুঝা কঠিন!

Error with source maps:
Error at Line 15 in main.ts ✓
Original file! বুঝতে সহজ!

Source map = compiled → original mapping
```

**CMD vs ENTRYPOINT:**
```dockerfile
CMD ["node", "dist/main"]
- Override করা যায়
- docker run myapp yarn test
  → CMD ignored, yarn test চলবে

ENTRYPOINT ["node"]
CMD ["dist/main"]
- docker run myapp yarn test
  → node yarn test (ভুল!)
- More fixed

Use CMD for flexibility ✓
```

## 🔒 Part 4: Security & Optimization

### Security Features:

```
1. Distroless Image:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

❌ Regular image:
- Shell access
- Package managers
- 100+ vulnerabilities

✓ Distroless:
- No shell
- No package managers
- <10 vulnerabilities

Hacker can't:
- cat /etc/passwd
- wget malware
- apt install tools

2. Non-root User:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

/home/node/app = node user owns
Not root! Limited permissions

3. Minimal Attack Surface:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Only what's needed to run:
✓ Node.js
✓ App code
✓ Prod dependencies

NOT included:
✗ Build tools
✗ Dev dependencies
✗ Test frameworks
✗ Compilers

Network analogy:
Firewall: শুধু needed ports খোলা
Docker: শুধু needed software রাখা
```

### Layer Caching Strategy:

```dockerfile
# ❌ Bad ordering:
COPY . .                    ← পরিবর্তন হয় প্রায়ই
RUN yarn install            ← Re-run হবে সবসময়!

# ✓ Good ordering (এই Dockerfile):
COPY package.json .         ← কম পরিবর্তন
RUN yarn install            ← Cached বেশিরভাগ সময়!
COPY . .                    ← শেষে copy

Timeline:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Build 1 (First time):
COPY package.json    → 0.1s
RUN yarn install     → 300s (5 min) ⏰
COPY dist            → 1s
Total: 301 seconds

Code change (main.ts edit)

Build 2 (With caching):
COPY package.json    → Cached (0s) ✓
RUN yarn install     → Cached (0s) ✓
COPY dist            → 1s (new build)
Total: 1 second! ⚡

Build time reduced 300x!
```

## 🌐 Part 5: বাস্তব উদাহরণ - পুরো Process

### Development থেকে Production:

```
Step 1: Code লেখা
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Developer Computer:
/my-project/
├── src/
│   ├── main.ts         ← TypeScript code
│   └── user.service.ts
├── package.json
├── tsconfig.json
└── Dockerfile          ← এই file!

Step 2: Docker Build
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Command:
$ docker build --build-arg PORT=3000 -t myapp:v1 .

Process:
[Stage 1: Builder]
→ FROM node:22-slim
→ COPY code
→ yarn install (all deps)
→ yarn build (TypeScript → JavaScript)
✓ Compilation done!

[Stage 2: Prod Builder]
→ FROM node:22-slim (fresh)
→ yarn install (prod only)
✓ Production deps ready!

[Stage 3: Final]
→ FROM distroless
→ COPY node_modules (from stage 2)
→ COPY dist (from stage 1)
✓ Image ready!

Output:
Image: myapp:v1 (280 MB)

Step 3: Local Test
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

$ docker run -p 8080:3000 myapp:v1

Container starts:
┌──────────────────────────┐
│ Container (myapp:v1)     │
│                          │
│ /home/node/app/          │
│ ├── node_modules/        │
│ ├── package.json         │
```

## 🎯 Part 6: Best Practices Checklist

### ✅ এই Dockerfile যা করছে (Good):

```
1. Multi-stage build
   ✓ Small final image (280 MB vs 990 MB)

2. Production dependencies only
   ✓ No dev tools in production

3. Distroless base
   ✓ Secure, minimal attack surface

4. Layer caching optimized
   ✓ package.json before code copy

5. Non-root user
   ✓ /home/node/app (secure)

6. Source maps enabled
   ✓ Better debugging

7. Clear stage names
   ✓ builder, prod-builder (readable)

8. ARG for flexibility
   ✓ PORT configurable at build time
```
