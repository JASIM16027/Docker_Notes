   ↓

   ↓          ↓

- `client` → React frontend

- `postgres` → Data store

## ২. POSTGRES — Database

```yaml
services:
  postgres:
    # Docker Hub থেকে official postgres image নামাও
    # latest = সবচেয়ে নতুন version (production এ specific version দেওয়া ভালো)
    image: "postgres:latest"

    environment:
      # Postgres এর default password set করো
      # এই password দিয়েই api connect করবে
      - POSTGRES_PASSWORD=postgres_password
      # Username default: postgres
      # Database default: postgres
      # Port default: 5432
```

> 💡 Postgres এর জন্য কোনো `volumes` নেই — মানে container বন্ধ হলে **data মুছে যাবে**। Production এ অবশ্যই volume দিতে হবে।

## ৪. NGINX — Gateway / Reverse Proxy

```yaml
  nginx:
    depends_on:
      - api       # api চালু হওয়ার পরে nginx চালু হবে
      - client    # client চালু হওয়ার পরে nginx চালু হবে
      # কারণ: nginx এদের কাছে traffic পাঠায়
      # এরা না উঠলে nginx এর কিছু করার নেই

    restart: always
    # Container crash করলে → automatically restart করো
    # nginx হলো সবার gateway, এটা বন্ধ থাকলে কিছুই কাজ করবে না

    build:
      dockerfile: Dockerfile.dev   # nginx এর custom config আছে
      context: ./nginx             # ./nginx/ folder এ Dockerfile আছে
      # Custom nginx.conf দিয়ে routing rules define করা আছে

    ports:
      - "3050:80"
      # বাম:  তোমার computer এর port 3050
      # ডান:  nginx container এর port 80
      # মানে: localhost:3050 → nginx container এর 80 port
```

### Nginx কী করে এখানে?

```
Browser: localhost:3050/api/values
         ↓
      nginx
         ↓ /api/* → api container এ পাঠাও
         ↓ /*     → client container এ পাঠাও

# nginx.conf এ এরকম থাকে:
location /api {
    proxy_pass http://api:5000;
}
location / {
    proxy_pass http://client:3000;
}
```

## ৫. API — Express Backend

```yaml
  api:
    build:
      dockerfile: Dockerfile.dev
      context: ./server      # ./server/ folder এ Dockerfile আছে

    volumes:
      - /app/node_modules
      # Container এর node_modules কে protect করো
      # Local এর node_modules দিয়ে overwrite হবে না
      # কারণ: container আর local OS আলাদা, modules different হতে পারে

      - ./server:/app
      # Local ./server/ folder → container এর /app/ এ sync করো
      # তুমি server/ এ code বদলালে → container এ সাথে সাথে reflect করবে
      # Hot reload কাজ করবে! restart লাগবে না

    environment:
      # Redis connection info
      - REDIS_HOST=redis      # "redis" = docker compose service এর নাম
      - REDIS_PORT=6379       # Redis এর default port

      # PostgreSQL connection info
      - PGUSER=postgres       # Postgres default username
      - PGHOST=postgres       # "postgres" = docker compose service এর নাম
                              # Docker এর internal DNS — IP লাগে না!
      - PGDATABASE=postgres   # Default database name
      - PGPASSWORD=postgres_password  # Postgres service এ যা দিয়েছিলাম
      - PGPORT=5432           # Postgres এর default port
```

> 💡 **PGHOST=postgres** — এটা IP address না! এটা service এর নাম।
> Docker Compose automatically internal DNS বানায়।
> `postgres` লিখলেই postgres container কে খুঁজে পাবে।

## ৭. WORKER — Background Job Processor

```yaml
  worker:
    build:
      dockerfile: Dockerfile.dev
      context: ./worker      # ./worker/ folder এ Dockerfile আছে

    volumes:
      - /app/node_modules    # node_modules protect
      - ./worker:/app        # local code sync

    environment:
      - REDIS_HOST=redis     # Redis service এ connect করো
      - REDIS_PORT=6379

      # Postgres connection নেই!
      # Worker শুধু Redis থেকে job নেয় আর result Redis এ রাখে
      # Database directly access করে না
```

### Worker কী করে?

```
Redis Queue:
  api → "calculate fibonacci(10)" push করলো
          ↓
  worker → queue থেকে নিলো
         → fib(10) calculate করলো
         → result Redis এ store করলো
          ↓
  api → Redis থেকে result নিয়ে client কে দিলো
```

## পুরো Flow একসাথে

```
তুমি browser এ localhost:3050 খুললে:

1. nginx পেলো request
   ↓
2. "/" → client (React app) দিলো
   "/api" → api (Express) দিলো

3. তুমি "10" submit করলে:
   client → POST /api/values → nginx → api

4. api:
   → postgres এ "10" save করলো
   → redis queue তে "calculate 10" push করলো

5. worker:
   → redis থেকে "10" পেলো
   → fib(10) = 55 calculate করলো
   → redis এ result save করলো

6. client GET /api/values করলে:
   → api postgres + redis থেকে data নিলো
   → client কে দিলো
   → browser এ দেখালো
```

### `rewrite /api/(.*) /$1 break;` — বিস্তারিত ব্যাখ্যা


# default.conf

```
upstream client {
    server client:3000;      # React container
}

upstream api {
    server api:5000;         # Express container
}

server {
    listen 80;

    # Rule 1: বাকি সব কিছু → React এ পাঠাও
    location / {
        proxy_pass http://client;
    }

    # Rule 2: /api দিয়ে শুরু হলে → Express এ পাঠাও
    location /api {
        rewrite /api/(.*) /$1 break;   # /api/values/all → /values/all
        proxy_pass http://api;
    }
}

```
এই line-টা বোঝার আগে একটা সমস্যা বুঝতে হবে।

---

### সমস্যাটা কী?

Browser request করে: `GET /api/values/all`

Nginx এটা Express-এ পাঠায়। কিন্তু Express server কিন্তু `/api/values/all` route চেনে না। সে শুধু চেনে `/values/all`।

কারণ Express নিজেই তো জানে সে একটা API server — তাকে আলাদা করে `/api` prefix দিয়ে route লেখার দরকার নেই। Express-এর code এরকম:

```js
app.get('/values/all', ...)      // ✅ Express এটা বোঝে
app.get('/api/values/all', ...)  // ❌ এভাবে লেখা নেই
```

তাহলে Nginx যদি সরাসরি `/api/values/all` পাঠায়, Express বলবে **404 Not Found।**

---

### `rewrite` দিয়ে সমাধান

```nginx
location /api {
    rewrite /api/(.*) /$1 break;
    proxy_pass http://api;
}
```

এই line টা `/api/` prefix কেটে ফেলে Express-এ পাঠায়।

**ভেঙে দেখি:**

| অংশ | মানে |
|---|---|
| `/api/(.*)` | `/api/` এর পরে যা আছে সেটা capture করো |
| `(.*)` | যেকোনো character, যেকোনো সংখ্যা — এটাই **capture group** |
| `/$1` | captured অংশটা `/` দিয়ে শুরু করে নতুন path বানাও |
| `break` | rewrite করার পরে আর কোনো rule দেখো না, এখনই forward করো |

**উদাহরণ:**

```
Browser পাঠায়:       /api/values/all
                         ↓
(.*)  capture করে:   values/all
                         ↓
/$1  বানায়:          /values/all
                         ↓
Express পায়:         /values/all   ✅
```

---

### `break` কেন দরকার?

Nginx-এ rewrite-এর দুটো flag আছে:

**`last`** — rewrite করার পরে আবার সব location rules থেকে শুরু করে match খোঁজে। এতে infinite loop হওয়ার সম্ভাবনা আছে।

**`break`** — rewrite করো, তারপর আর কোনো rule দেখো না। সরাসরি এই location-এই থাকো এবং forward করো। এটাই safe এবং এখানে সঠিক।

---

### পুরো flow একবার দেখি

```
Browser
  │  GET /api/values/all
  ▼
Nginx
  │  "/api" match হলো
  │  rewrite → /values/all
  │  break → আর rule দেখবো না
  ▼
Express Server
  │  GET /values/all    ← prefix ছাড়া
  ▼
app.get('/values/all') ✅ match!
  │
  ▼
JSON response → Nginx → Browser
```

---

### না দিলে কী হতো?

```nginx
location /api {
    proxy_pass http://api;   # rewrite নেই
}
```

Express পেতো `/api/values/all` — যে route সে চেনে না → **404 error।**

সংক্ষেপে বলতে গেলে, `rewrite` হলো একটা **path translator** — browser-এর ভাষা (`/api/values/all`) Express-এর ভাষায় (`/values/all`) অনুবাদ করে দেয়।



## Dockerfile — লাইন বাই লাইন বাংলায় 🐳

---

### এটা কোন service এর Dockerfile?

আগের Docker Compose দেখলে এটা **client** service এর Dockerfile — কারণ:
- `node:16-alpine` use করছে
- `/home/node/app` path (compose file এ client এর volume এই path এ ছিল)
- `npm start` দিয়ে React app চালু করে

## লাইন বাই লাইন

---

### Line 1 — Base Image

```dockerfile
FROM node:16-alpine
```

**কী করে:**
```
Docker Hub থেকে node:16-alpine image নামাও
এটাই তোমার container এর ভিত্তি

node:16        → Node.js version 16
alpine         → Alpine Linux (অনেক ছোট OS, মাত্র ~5MB)
```

**কেন alpine?**
```
node:16         → ~900MB  (Ubuntu based, অনেক বড়)
node:16-alpine  → ~120MB  (Alpine based, অনেক ছোট)

Production এ image ছোট হলে:
  ✅ Download দ্রুত হয়
  ✅ Security surface কম
  ✅ Storage কম লাগে
```

---

### Line 2 — User পরিবর্তন

```dockerfile
USER node
```

**কী করে:**
```
এখন থেকে সব command "node" user হিসেবে চলবে
(আগে ছিল "root" user)
```

**কেন root user use করবো না?**
```
Default এ Docker container root হিসেবে চলে।
Root মানে:
  ❌ সব permission আছে
  ❌ Hack হলে পুরো system access পাবে

"node" user মানে:
  ✅ Limited permission
  ✅ Security ভালো
  ✅ Best practice

Alpine node image এ "node" নামে একটা
non-root user already আছে।
```

---

### Line 3 — Directory তৈরি করো

```dockerfile
RUN mkdir -p /home/node/app
```

**কী করে:**
```
Container এর ভেতরে /home/node/app folder বানাও

mkdir     → directory তৈরি করো
-p        → parent folder না থাকলে সেটাও বানাও
            (home/ না থাকলে home/ বানাও, তারপর node/, তারপর app/)
```

**কেন এই path?**
```
USER node বলেছি → node user এর home = /home/node/
সেখানে app/ folder বানাচ্ছি
node user এই folder এ write করার permission আছে ✅
```

---

### Line 4 — Working Directory set করো

```dockerfile
WORKDIR /home/node/app
```

**কী করে:**
```
এখন থেকে সব command এই folder এ চলবে
মানে: cd /home/node/app এর মতো
```

**উদাহরণ:**
```
WORKDIR ছাড়া:
  RUN npm install  → কোথায় চলবে? অজানা!

WORKDIR দিয়ে:
  RUN npm install  → /home/node/app এ চলবে ✅

পরের সব COPY, RUN, CMD
এই folder কে base হিসেবে ধরবে।
```

---

### Line 5 — package.json copy করো

```dockerfile
COPY --chown=node:node ./package.json ./
```

**কী করে:**
```
Local machine এর ./package.json
→ Container এর /home/node/app/package.json এ copy করো

--chown=node:node → file এর owner হবে node user
                    (root নয়, কারণ আমরা node user use করছি)
```

**কেন পুরো code আগে না করে শুধু package.json?**
```
Docker প্রতিটা line কে Layer হিসেবে cache করে।

যদি সব একসাথে copy করতাম:
  COPY ./ ./          → code বদলালে
  RUN npm install     → আবার npm install! 😫 (ধীর)

আলাদা করলে:
  COPY package.json   → Layer 1 (cache)
  RUN npm install     → Layer 2 (cache, package.json না বদলালে)
  COPY ./ ./          → Layer 3 (শুধু এটা rebuild)

Code বদলালে শুধু Layer 3 rebuild হয়।
npm install আর করতে হয় না! ⚡ দ্রুত!
```

---

### Line 6 — Dependencies install করো

```dockerfile
RUN npm install
```

**কী করে:**
```
package.json দেখে সব dependencies install করো
/home/node/app/node_modules/ folder তৈরি হবে
```

**Cache কীভাবে কাজ করে:**
```
প্রথমবার build:
  package.json copy হলো
  npm install → সব package download (২-৩ মিনিট)
  এই layer cache হলো

দ্বিতীয়বার build (শুধু code বদলেছে):
  package.json same → cache hit! ✅
  npm install → skip! (cache থেকে নেয়)
  সময়: ১০ সেকেন্ড!

package.json বদলালে:
  cache miss → npm install আবার চলবে
```

### Line 7 — বাকি সব code copy করো

### Line 8 — Container চালু হলে কী করবে

```dockerfile
CMD ["npm", "start"]
```

**কী করে:**
```
Container start হলে এই command চালাও:
  npm start → package.json এর "start" script চালায়

```

→ Runtime এ চলে (container চালু হলে)
→ প্রতিবার container start এ চলে
→ শুধু একটাই CMD থাকতে পারে
```

Step 2: USER node
  → এখন থেকে "node" user

Step 3: RUN mkdir -p /home/node/app
  → folder বানাও

Step 4: WORKDIR /home/node/app
  → এই folder এ কাজ করবো

Step 5: COPY package.json ./
  → package.json নিয়ে আসো

Step 6: RUN npm install
  → সব packages install করো (cache হবে)

Step 7: COPY ./ ./
  → বাকি সব code নিয়ে আসো

Image ready! ✅

docker run করলে:
  CMD: npm start → React app চালু → port 3000
```

| Lightweight base image | `alpine` use |

### Root vs Node User — বিস্তারিত ব্যাখ্যা

---

### প্রথমে বুঝি: Linux-এ User কী?

Linux-এ প্রতিটা process কোনো না কোনো **user** হিসেবে চলে। সেই user-এর যতটুকু permission আছে, process ততটুকুই করতে পারে।

Docker container আসলে একটা Linux process — তাই এখানেও user matter করে।

---

### Root User কী?

Root হলো Linux-এর **superuser** — সে সব কিছু করতে পারে।

```
root পারে:
  ✅ যেকোনো file read/write/delete করতে
  ✅ যেকোনো port এ listen করতে
  ✅ যেকোনো system command চালাতে
  ✅ অন্য user তৈরি/delete করতে
  ✅ Network config বদলাতে
  ✅ Host machine এর file access করতে (volume mount থাকলে)
```

Docker-এ default হলো root। মানে তুমি যদি `USER` না লেখো, সব কিছু root হিসেবে চলে।

---

### সমস্যাটা কোথায়?

ধরো তোমার React container hack হলো — কেউ malicious code inject করলো।

**Root হিসেবে চললে:**
```
হ্যাকার পেলো root access
  → /etc/passwd বদলাতে পারবে
  → নতুন backdoor তৈরি করতে পারবে
  → Volume mount এর মাধ্যমে host machine এ ঢুকতে পারবে
  → অন্য container এ attack করতে পারবে
  → পুরো server এর control নিতে পারবে 😱
```

**Node user হিসেবে চললে:**
```
হ্যাকার পেলো সীমিত access
  → শুধু /home/node/app এর file দেখতে পারবে
  → system file ছুঁতে পারবে না
  → host machine এ যেতে পারবে না
  → ক্ষতি অনেক কম 😌
```

---

### Node User এর Permission কতটুকু?

```
node user পারে:
  ✅ /home/node/app এ read/write করতে
  ✅ npm install চালাতে
  ✅ React app run করতে
  ✅ Port 3000 এ listen করতে (1024 এর উপরে)

node user পারে না:
  ❌ /etc/ system file বদলাতে
  ❌ Port 80 এ directly listen করতে (1024 এর নিচে = root only)
  ❌ অন্য user এর file access করতে
  ❌ System package install করতে (apt-get, apk)
```

এই কারণেই Nginx port 80 এ থাকে — node user সেটা পারে না, root লাগে। কিন্তু React app port 3000 এ চলে, সেটা node user পারে।

---

### Dockerfile এ কীভাবে কাজ করে?

```dockerfile
FROM node:16-alpine
# এখানে user = root (default)
# root হিসেবে system কাজ করা যায়

USER node
# এখান থেকে user = node
# সব command এখন node হিসেবে চলবে

RUN mkdir -p /home/node/app
# node user এই folder বানাচ্ছে
# তাই folder এর owner = node ✅

COPY --chown=node:node ./package.json ./
# --chown না দিলে file এর owner হতো root
# root এর file node user read করতে পারতো না ❌
# --chown=node:node দিলে owner = node ✅
```

---

### `--chown` কেন দরকার?

```
USER node বললেই সব ঠিক হয়ে যায় না।

COPY করলে file এর owner হয় যে COPY চালাচ্ছে।
কিন্তু COPY সবসময় root হিসেবে চলে!

তাই --chown=node:node না দিলে:
  file copy হবে ✅
  কিন্তু owner = root ❌
  node user সেই file write করতে পারবে না ❌
  npm install fail করবে ❌

--chown=node:node দিলে:
  file copy হবে ✅
  owner = node ✅
  node user সব কাজ করতে পারবে ✅
```

---

### এক নজরে পার্থক্য

| বিষয় | Root | Node User |
|---|---|---|
| সব file access | ✅ | ❌ |
| System config বদলানো | ✅ | ❌ |
| Port 80 use | ✅ | ❌ |
| Port 3000 use | ✅ | ✅ |
| App চালানো | ✅ | ✅ |
| Hack হলে ক্ষতি | পুরো server | শুধু app folder |
| Best practice | ❌ | ✅ |

---

### মূল কথা

Container-এ app চালাতে root-এর power দরকার নেই। যতটুকু permission দরকার ততটুকুই দাও — এটাই **Principle of Least Privilege**। Node user দিয়ে React app সব কাজ করতে পারে, কিন্তু hack হলেও ক্ষতি অনেক সীমিত থাকে।



Terminal-এ এই command দাও:

```bash
docker exec -it e90da0cffd05 sh
```

এটা container-এর ভেতরে ঢুকিয়ে দেবে। তারপর সাধারণ Linux command দিয়ে folder দেখতে পারবে:

```bash
ls          # current folder-এর files দেখো
ls -la      # hidden files সহ details দেখো
pwd         # এখন কোন folder-এ আছো
cd /etc/nginx/conf.d    # nginx config folder-এ যাও
cat default.conf        # config file দেখো
```

**বের হতে চাইলে:**
```bash
exit
```

---

**যদি `sh` কাজ না করে** (অনেক container-এ `sh` থাকে না), তাহলে:

```bash
docker exec -it e90da0cffd05 bash
```

**Container-এ না ঢুকে সরাসরি দেখতে চাইলে:**

```bash
docker exec e90da0cffd05 ls /etc/nginx/conf.d
```
