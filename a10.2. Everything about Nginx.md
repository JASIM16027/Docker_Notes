— Nginx নিজেই একটা Load Balancer হতে পারে।

---

### এক লাইনে বলতে গেলে

**Load Balancer** একটা **concept** — traffic ভাগ করে দেওয়ার ধারণা।

**Nginx** একটা **tool** — যেটা এই concept implement করতে পারে, আবার আরো অনেক কিছুও করতে পারে।

---

### Load Balancer কী করে?

শুধু একটাই কাজ — traffic ভাগ করা।

```
১০০০ request আসলো
     │
  Load Balancer
  ├── Server 1 → ৩৩৩ request
  ├── Server 2 → ৩৩৩ request
  └── Server 3 → ৩৩৪ request
```

কোনো server down হলে সেখানে আর পাঠাবে না। এটুকুই।

---

### Nginx কী কী করতে পারে?

```
Nginx = Load Balancer
      + Reverse Proxy
      + Static File Server
      + SSL Termination
      + Rate Limiting
      + Caching
      + Compression
```

Load balancing তার অনেক feature এর একটা মাত্র।

---

### Nginx দিয়ে Load Balancing

```nginx
upstream express {
    # Default: Round Robin
    # প্রতিটা request একটার পর একটা server এ যাবে
    server express-1:5000;
    server express-2:5000;
    server express-3:5000;
}

server {
    location /api {
        proxy_pass http://express;
    }
}
```

এটুকু লিখলেই Nginx load balancer হিসেবে কাজ করে।

---

### Load Balancing এর ধরন

**Round Robin** (default)
```
Request 1 → Server 1
Request 2 → Server 2
Request 3 → Server 3
Request 4 → Server 1  ← আবার শুরু
```

**Least Connections**
```nginx
upstream express {
    least_conn;   # যে server এ কম connection সেখানে পাঠাও
    server express-1:5000;
    server express-2:5000;
}
```

**IP Hash** — একই user সবসময় একই server এ যাবে
```nginx
upstream express {
    ip_hash;      # user এর IP দেখে সবসময় same server এ
    server express-1:5000;
    server express-2:5000;
}
```

**Weight** — কোনো server বেশি powerful হলে
```nginx
upstream express {
    server express-1:5000 weight=3;  # ৩ গুণ বেশি traffic পাবে
    server express-2:5000 weight=1;
}
```

### তাহলে AWS/GCP এর Load Balancer আলাদা কেন?

Cloud এর Load Balancer গুলো যেমন AWS ALB, GCP Load Balancer — এগুলো **dedicated hardware/managed service।**

| | Nginx | AWS ALB |
|---|---|---|
| Setup | নিজে করতে হয় | Managed, AWS করে দেয় |
| Cost | Free | Pay per use |
| Scale | একটা server এর মধ্যে | Unlimited, global |
| Maintenance | নিজে করতে হয় | AWS করে |
| DDoS protection | Limited | Built-in |

### কখন কোনটা?

```

বড় production, cloud এ

  (দুটো একসাথে ব্যবহার হয়)
```

### Nginx এর সব Features — বিস্তারিত

---

### ১. Load Balancer

Traffic ভাগ করে একাধিক server এ পাঠায়।

```nginx
upstream express {
    server express-1:5000;
    server express-2:5000;
    server express-3:5000;
}
```

```
১০০০ request
      │
    Nginx
    ├── express-1 → ৩৩৩টা
    ├── express-2 → ৩৩৩টা
    └── express-3 → ৩৩৪টা
```

একটা server down হলে Nginx বাকিদের মধ্যে traffic পাঠায়। কোনো request হারায় না।

---

### ২. Reverse Proxy

Client কখনো জানে না পেছনে কোন server আছে। Nginx মাঝখানে দাঁড়িয়ে সব handle করে।

```
Client জানে:      myapp.com
আসলে যাচ্ছে:     192.168.1.10:3000 (React)
                  192.168.1.11:5000 (Express)
```

```nginx
location / {
    proxy_pass http://react:3000;
}
location /api {
    proxy_pass http://express:5000;
}
```

তুমি এটা আগের diagram এ দেখেছো — এটাই reverse proxy।

---

### ৩. Static File Server

HTML, CSS, JS, Image — এই static file গুলো Nginx নিজেই serve করতে পারে। Express/React server এর কাছে যেতেই হয় না।

```nginx
location / {
    root /var/www/html;     # এই folder এ file আছে
    index index.html;
    try_files $uri $uri/ /index.html;
}
```

```
Client: GET /logo.png
Nginx: /var/www/html/logo.png আছে? → সরাসরি পাঠিয়ে দিলাম ✅
Express কে জিজ্ঞেসই করলাম না!
```

**কেন দরকার:**
```
Express দিয়ে static file serve করলে:
  Request → Express (Node.js process) → file read → response
  Node.js এর CPU/Memory খরচ হয়

Nginx দিয়ে করলে:
  Request → Nginx → file read → response
  অনেক দ্রুত, Express এর উপর চাপ নেই ✅
```

---

### ৪. SSL Termination

HTTPS মানে সব data encrypted হয়ে আসে। কিন্তু decrypt করা CPU-intensive কাজ। Nginx এই কাজটা একবার করে, তারপর ভেতরে plain HTTP পাঠায়।

```
Browser
  │  HTTPS (encrypted) 🔒
  ▼
Nginx ← এখানে decrypt হয়
  │  HTTP (plain) 
  ▼
React/Express Server
```

```nginx
server {
    listen 443 ssl;                          # HTTPS port

    ssl_certificate     /etc/ssl/cert.pem;   # Certificate
    ssl_certificate_key /etc/ssl/key.pem;    # Private key

    location / {
        proxy_pass http://react:3000;        # ভেতরে plain HTTP
    }
}

# HTTP → HTTPS redirect
server {
    listen 80;
    return 301 https://$host$request_uri;
}
```

**কেন এটা ভালো:**
```
SSL ছাড়া:
  প্রতিটা server এ আলাদা SSL certificate লাগতো
  React server এ SSL + Express server এ SSL = জটিল

SSL Termination দিয়ে:
  শুধু Nginx এ একটা certificate ✅
  ভেতরের server গুলো SSL জানেই না
  সহজ, কম খরচ
```

---

### ৫. Rate Limiting

একটা user বা IP থেকে বেশি request আসলে block করো। DDoS attack বা brute force থেকে বাঁচায়।

```nginx
# প্রতি IP থেকে প্রতি সেকেন্ডে max ১০টা request
limit_req_zone $binary_remote_addr zone=mylimit:10m rate=10r/s;

server {
    location /api {
        limit_req zone=mylimit burst=20;
        proxy_pass http://express;
    }
}
```

```
Normal user:
  ১০ request/second → ✅ allow

Bot/Attacker:
  ১০০০ request/second → ❌ block, 429 Too Many Requests
```

**`burst=20` মানে:**
```
হঠাৎ ২০টা request একসাথে আসলে allow করো
কিন্তু তারপর rate limit enforce করো
(Normal user এর হঠাৎ বেশি request হতে পারে)
```

---

### ৬. Caching

একই request বারবার আসলে server কে না জিজ্ঞেস করে Nginx নিজেই জমানো response দেয়।

```nginx
# Cache storage define করো
proxy_cache_path /tmp/nginx_cache levels=1:2
                 keys_zone=my_cache:10m
                 max_size=1g
                 inactive=60m;

server {
    location /api {
        proxy_cache my_cache;
        proxy_cache_valid 200 10m;    # 200 response ১০ মিনিট cache করো
        proxy_pass http://express;
    }
}
```

```
প্রথম request:
  Client → Nginx → Express → Database
  Response: cache এ save হলো

পরের ১০ মিনিট একই request:
  Client → Nginx → cache থেকে দিলো ✅
  Express আর Database কে জিজ্ঞেসই করলো না!
```

**কোন জিনিস cache করা উচিত:**
```
✅ Cache করো:
  Product list (ঘন ঘন বদলায় না)
  Blog posts
  Static API responses

❌ Cache করো না:
  User এর personal data
  Payment information
  Real-time data (stock price, live score)
```

---

### ৭. Compression

Response পাঠানোর আগে Nginx সেটা compress করে। File size ছোট হয় → দ্রুত পৌঁছায়।

```nginx
gzip on;
gzip_types text/plain text/css application/json application/javascript;
gzip_min_length 1000;      # ১KB এর বড় file compress করো
gzip_comp_level 6;         # Compression level (1-9, বেশি = ধীর কিন্তু ছোট)
```

```
Compression ছাড়া:
  index.js → 500KB → Browser এ পাঠাও

Compression দিয়ে:
  index.js → gzip → 150KB → Browser এ পাঠাও
  Browser নিজে decompress করে নেয়

৭০% কম data transfer! ✅
```

**কী কী compress হয়:**
```
✅ ভালো compress হয়:
  HTML, CSS, JS (text based) → ৬০-৮০% ছোট হয়
  JSON API response → ৭০-৯০% ছোট হয়

❌ Compress করার দরকার নেই:
  Image (JPG, PNG) → already compressed
  Video → already compressed
  ZIP file → already compressed
```

### সব একসাথে কাজ করলে

```
Browser: GET https://myapp.com/api/products

Step 1 — SSL Termination:
  HTTPS decrypt করলো Nginx

Step 2 — Rate Limiting:
  এই IP থেকে limit এর মধ্যে? ✅ allow

Step 3 — Caching:
  /api/products cache এ আছে?
  হ্যাঁ → সরাসরি দিলো, Express এ গেলোই না ✅
  না  → পরের step

Step 4 — Load Balancing:
  Express ৩টা container এর মধ্যে কোনটায় কম load?
  → express-2 তে পাঠাও

Step 5 — Reverse Proxy:
  express-2:5000 এ forward করলো

Step 6 — Compression:
  Response gzip করলো (500KB → 150KB)

Browser: data পেলো 🎉
```

একটাই Nginx এতগুলো কাজ করলো — Express কে শুধু business logic নিয়ে ভাবতে হলো।
